    # Map

    JDK1.8Map的了解

    Map是java里边的一个接口，常见的实现类有HashMap、LinkedHashMap、TreeMap和ConcurrentHashMap

    HashMap底层数据结构是数组+链表/红黑树

    LinkedHashMap底层数据结构是数组+链表+双向链表

    TreeMap底层数据结构是红黑树

    ConcurrentHashMap底层数据结构是数组+链表/红黑树

    -------------------------------------------------

    # HashMap

    创建new（HashMap有几个构造方法，最主要的是 指定初始大小 和 负载因子大小）

    初始大小

    如果我们不指定hashmap的大小，默认HashMap的大小为16，负载因子的大小为0.75

    HashMap的大小只能是2次幂

    假设传一个10进去，实际上最终HashMap大小为16，传7进去，HashMap最终大小为8，（即1.2.4.8.16.……）可以通过tableSizeFor看到

    为什么HashMap大小是2次幂？

    我们把元素添加进HashMap的时候，需要算出这个元素在数组的位置（也就是Hash)，

    在HashMap里面用的是位运算来代替取模（位运算效率要高于取余的运算）,大小为2次幂，才能合理用位运算替代取模

    负载因子

    负载因子的大小决定着哈希表的扩容和哈希冲突

    比如默认HashMap大小为16，负载因子0.75，所以一旦超过12个元素，哈希表就需要扩容

    扩容

    每次put元素进去的时候，都会检查HashMap大小有没有超过这个阈值，如果超过，就需要扩容

    因为HashMap大小是2次幂，所以扩容的时候是扩容原来的2倍

    扩容会调用resize()方法 ， 会遍历一遍所有的元素，要重新分配一次下标值， 尽量避免

    不过不需要重新计算hash值，会多一个bit位，要么就是分配在原来的位置，要么就是原位置+原来的容量位置

    因为扩容是耗时的，那么能不能把负载因子调高，比如1的时候才扩容？

    可以是可以，但不推荐，因为负载因子高了，哈系冲突的概率也会增加，会导致查找速度变慢，也会增加耗时

    传递的key怎么算哈希值？

    这样做的好处增加随机性，减少哈希碰撞的可能性

    put()

    1、先通过 hash 值计算出 下标 ；

    2、如果没有碰撞冲突，则直接插入；

    3、如果出现碰撞冲突了，则需要处理冲突：

    a 如果该桶使用红黑树处理冲突，则调用红黑树的方法插入数据；

    b 否则采用传统的链式方法插入。如果链的长度达到临界值，则把链转变为红黑树；

    数组大小大于64且链表的大小大于8的时候 将链表改为红黑树

    4、如果桶中存在重复的键，则为该键替换新值 value；

    5、如果 size 大于阈值 容量*扩容因子 ，则进行扩容resize()；

    hash值 & (数组length - 1) 有什么好处？

    最终目的让碰撞的几率降低，分布均匀

    因为数组长度是2次幂，length - 1 之后 二进制表示都是除了第一位，后面都是1，进行位运算的时候

    get()

    get的时候，计算出该key所在的index，然后判断是否有哈希冲突

    假如没有hash冲突就直接返回，

    如果有冲突就用equals判断 ，相等则返回

    HashMap怎么判断一个元素是否相同？

    先比较下标值，随后用==运算符和equals()来判断元素是否相同

    （hash相同说明，哈希冲突了，还得用==或者equals()来判断是否相同）

    什么时候转换为红黑树？

    数组大小大于64且链表的大小大于8的时候 将链表改为红黑树，当红黑树大小为6，退化为链表

    链表和红黑树的转换主要是 对 查询和插入性能的考虑

    链表插入O(1) 查询O(N) ； 红黑树查询插入都是O(logN)；

    解决hashQ冲突 (哈希冲突) 有以下四种方法：

    • 1.链地址法 hashmap
    • 2.再哈希法 hash冲突，用第二个哈希算法
    • 3.建立公共溢出区 将哈希表分为基本表和溢出表两部分，凡是和基本表发生冲突的元素，一律填入溢出表。
    • 4.开放定址法 冲突，京 就找另一个不冲突的下标 三种方法1、找下一个下标。2、左右跳跃式探测。3、随机数

    -------------------------------------------------

    Hashtable和HashMap

    初始容量不同：HashMap的初始容量为16，Hashtable为11，负载因子都是0.75

    扩容机制不同：HashMap是翻2倍，Hashtable是翻两倍+1

    Hashtable是线程安全的，不允许为null

    HashMap是非线程安全,允许为null,但 null 作为键只能有一个，null 作为值可以有多个

    --------------------------------------.-----------

    # LinkedHashMap

    底层是数组+链表+双向链表，继承了HashMap，在HashMap的基础上加了一个双向链表

    有了双向链表，可以保证插入有序

    -------------------------------------------------

    # TreeMap

    底层是红黑树，key不能为null（为null就不能排序了）

    会对key进行排序，通过Comparator来进行比较

    ------------------------------------------

    HashMap还是TreeMap

    如果需要排序就用TreeMap

    -------------------------------------

    说一下 HashSet 的实现原理？

    HashSet底层由HashMap实现

    HashSet的值存放于HashMap的key上

    HashMap的value统一为PRESENT

    为什么HashMap不安全？

    在JDK1.8中，在并发执行put操作时会发生数据覆盖的情况。

    两个线程同时put操作的时候，

    A线程执行完第6行，判断了当前索引是空的，时间片耗尽被挂起

    B线程计算完索引是同一个位置，发现也是空的，就把数据put进来

    这时候A线程获得时间片，由于之前已经判断过hash碰撞的问题，就直接插入数据，就会把B线程插入的数据覆盖

    -------------------------------------------------

    解决并发问题可以用HashTable或者Collections.synchronizedMap，但是效率不高，

    所以要用concurrentHashMap

    # 线程安全ConcurrentHashMap

    HashMap不是线程安全的，Map的线程安全一般使用ConcurrentHashMap

    线程安全还有Hashtable和用Collections工具类来变成一个线程安全的Map

    但无论是Hashtable还是Collections包装出来的都比较低效（因为是直接在外层使用了synchronize）

    所以需要线程安全的时候都是使用ConcurrentHashMap

    https://blog.csdn.net/singwhatiwanna/article/details/104586114/

    ConcurrentHashMap的底层数据结构是数组+链表/红黑树。

    1.7 ： 对数组进行了分段（Segment），每个锁只锁其中一段，多线程访问容器的时候，就不会存在竞争，提高并发访问率

    1.8 ：

    取消了Segment分段锁

    通过在部分加锁synchronized和CAS算法来实现同步，synchronized 只锁定当前链表或红黑二叉树的首节点，这样只要 hash 不冲突，就不会产生并发，效率又提升 N 倍。

    get时候不加锁,Node都用了volatile修饰

    扩容时候会给每个线程分配对应的区间，为了防止putVal导致数据不一样，会给线程所负责的区间加锁

    JDK1.7和1.8 HashMap和ConcurrentHashMap区别

    HashMap 1.7 -> 1.8

    1.7的HashMap在扩容时候是头插法，1.8是尾插法，

    1.7HashMap还没有红黑树

    HashMap

    在1.7版本中，并发场景下 有以下问题：

    数据重复；

    死循环

    数据丢失

    1.8版本的HashMap解决了两大问题：

    死循环的问题

    查询效率低的问题（通过红黑树）

    ConcurrentHashMap 1.7 ->1.8

    **锁方面：**分段锁（Segment） 升级为 CAS + synchronized实现

    **问：**synchronized性能不好，为啥1.8还要多了synchronized？

    **答：**synchronized之前一直是重量级的锁，性能很差，但是到了jdk6的时候，对synchronized进行了优化。

    synchronized获取锁 是以锁升级的方式，先使用偏向锁优先同一线程，然后再次获取锁，如果失败，就升级为CAS轻量级锁，如果失败就会短暂自旋，防止线程被系统挂起。以上都失败就升级为重量级锁。是一步一步升级上去的，最初也是通过很多轻量级的方式锁定

    （偏向锁->CAS轻量级锁->自旋->重量级锁）

    **数据结构方面：**将Segment变成Node，减小了锁的粒度，使每个Node独立，原来二并发度16变成了每个Node都独立，提高了并发度。

    **hash冲突：**1.7采用链表存储，1.8先使用链表存储，满足条件就用红黑树存储

    **查询复杂度：**1.7链表O（N） 1.8转化为红黑树后O（logN）

    红黑树特性

    性质1：每个节点要么是红色，要么是黑色。

    性质2：根节点永远是黑色的。

    性质3：所有的叶节点都是空节点（即 null），并且是黑色的。

    性质4：每个红色节点的两个子节点都是黑色。（从每个叶子到根的路径上不会有两个连续的红色节点）

    性质5：从任一节点到其子树中每个叶子节点的路径都包含相同数量的黑色节点。
